# Дорожная карта: наша версия платформы агентов

Упрощённый аналог Sim с **нашими фичами**: создание агентов по промпту и короткий путь до деплоя (на выходе — Python-проект + Docker).

---

## Наши фичи (что делаем по-своему)

| Фича | Описание |
|------|----------|
| **Генерация по промпту** | Пользователь вводит описание цели → встроенный ИИ-агент предлагает/генерирует workflow (граф нод), можно править в редакторе. |
| **Выход = код + контейнер** | По кнопке «Экспорт» получаем Python-проект (структура, зависимости, логика шагов) + Dockerfile; пользователь деплоит сам (свой репо, свой сервер/k8s). |
| **Простой путь до деплоя** | Не привязка к облаку платформы: собрал сценарий → экспорт → `docker build && docker run` (или CI/CD по своему репо). |

Общее с Sim: визуальный редактор workflow (ReactFlow), блоки/ноды, идея «агенты + инструменты». Не делаем: их облако, их выполнение (Trigger.dev, E2B), полный набор интеграций на старте.

---

## Что берём за основу

- **Текущий flow-app** — уже есть редактор (React Flow), типы нод (Trigger, Action, Code), сохранение/загрузка JSON. Расширяем типами «Агент», при необходимости — подблоки (поля) в нодах.
- **Идеи из Sim:** разделение «блок (нода) ↔ шаг в пайплайне», граф как DAG для порядка выполнения, единый формат сериализации workflow (JSON) для генератора.

---

## Целевая структура репозитория

```
AI_Arch/
├── flow-app/              # Фронт (Vite + React + React Flow) — уже есть
│   └── расширяем: панель «Создать по промпту», тип ноды «Агент», кнопка «Экспорт в Python»
├── backend/               # Небольшой бэкенд (API для ИИ и генерации)
│   ├── api/
│   │   ├── generate_workflow.py   # POST: prompt → workflow JSON (ИИ)
│   │   └── export_python.py      # POST: workflow JSON → zip (Python + Dockerfile)
│   ├── generator/         # Шаблоны и логика кодогенерации (граф → Python)
│   └── requirements.txt
├── ROADMAP.md             # этот файл
├── sim-study.md           # конспект по Sim
└── README.md              # общее описание проекта (при желании)
```

Фронт может работать без бэкенда (только редактор + сохранение JSON в файл). Бэкенд нужен для: (1) генерации workflow по промпту, (2) экспорта в Python + Docker.

## Фазы разработки

### Фаза 1 — Укрепляем редактор (1–2 недели)

- [ ] Добавить тип ноды **«Агент»** (LLM-вызов: модель, системный промпт, вход/выход).
- [ ] Унифицировать формат workflow JSON (nodes + edges + описание полей блоков), чтобы бэкенд и генератор опирались на одну схему.
- [ ] В сайдбаре: кнопка **«Создать по описанию»** → поле ввода промпта → (пока заглушка) «Отправлять на бэкенд» или мок: по кнопке подставлять пример графа из JSON.

### Фаза 2 — Бэкенд и генерация по промпту (2–3 недели)

- [ ] Поднять минимальный **backend** (FastAPI или Node): один маршрут `POST /api/generate-workflow`, тело `{ "prompt": "..." }`, ответ `{ "nodes", "edges" }`.
- [ ] Реализовать вызов LLM (OpenAI/Anthropic/локальная модель): по промпту возвращать структурированный JSON графа (ноды + рёбра + минимальные data для нод). Structured output / JSON mode.
- [ ] На фронте: по кнопке «Создать по описанию» отправлять запрос, подставлять полученный граф в редактор.

### Фаза 3 — Экспорт в Python + Docker (2–3 недели)

- [ ] В бэкенде: маршрут `POST /api/export-python`, тело — workflow JSON, ответ — zip-архив.
- [ ] **Генератор:** граф → топологический порядок (DAG) → для каждого шага генерировать фрагмент Python (триггер = entrypoint, действие = вызов функции, агент = вызов LLM, код = кусок кода). Один общий `main.py` или модули по шагам + `requirements.txt` + `Dockerfile`.
- [ ] На фронте: кнопка **«Экспорт в Python»** → отправка текущего графа на API → скачивание zip.

### Фаза 4 — Полировка и опции (по мере необходимости)

- [ ] Тесты для генератора (фиксированный JSON → ожидаемый набор файлов).
- [ ] Опционально: выбор рантайма (например, Celery vs простой скрипт) при экспорте.
- [ ] Документация: как запустить бэкенд, как пользоваться «Создать по промпту» и «Экспорт в Python».

## Риски и упрощения

- **Качество графа от ИИ:** начать с жёсткой схемы (мало типов нод, фиксированные поля) и structured output; дорабатывать промпты и валидацию по мере тестов.
- **Сложность генератора:** первый вариант — линейный пайплайн (цепочка шагов без ветвлений); затем добавить ветвления по условию и параллельные ветки.
- **Аутентификация:** в первой версии не делать (все локально или один общий API key для ИИ). При необходимости позже — простой API key или OAuth.

## Итог

Да, **можно сделать свою версию на основе этих идей**: опираемся на текущий flow-app и конспект Sim, добавляем бэкенд с «промпт → workflow» и «workflow → Python + Docker», идём по фазам. Начинаем с укрепления редактора и формата JSON, затем подключаем ИИ и экспорт.
